# 贡献指南

感谢你考虑为 **框框·高考志愿填报顾问 Skill** 做出贡献！

这个项目的目标是将框框的志愿填报方法论蒸馏为可执行的AI Skill，帮助更多家境普通的学生科学填报志愿。我们欢迎所有形式的贡献。

---

## 🌟 贡献方式

### 1. 更新素材

如果你发现框框有新的志愿填报相关视频，或者政策有新变化：

**步骤**：
1. 从B站下载视频并提取字幕（SRT格式）
2. 将字幕文件放入 `references/sources/transcripts/`
3. 在字幕文件头部注明：
   - 视频标题
   - 发布日期
   - B站链接
   - BV号
4. 提交PR，说明新增素材的主题和价值

**示例提交信息**：
```
feat: 添加2026年最新志愿填报政策解读字幕

- 视频：《2026高考志愿填报新变化》
- 发布日期：2026-05-20
- BV号：BV1xxxxxxxxxx
- 主要内容：新高考改革后的填报策略调整
```

### 2. 改进方法论

如果你发现决策框架有不足，或总结出新的心智模型：

**适合改进的内容**：
- 发现现有心智模型的逻辑漏洞
- 提炼出新的决策启发式规则
- 补充缺失的场景判断逻辑
- 优化表达方式，使其更易理解

**不适合改进的内容**：
- 与框框核心理念冲突的观点
- 过度复杂化的理论框架
- 商业化导向的建议

**步骤**：
1. 在 `references/research/` 目录下创建新的调研文件（如需要）
2. 修改 `SKILL.md`，在相应章节添加或修改内容
3. 在PR中详细说明：
   - 发现了什么问题
   - 为什么需要改进
   - 改进后如何解决问题
   - 是否经过实际对话测试验证

**示例提交信息**：
```
improve: 补充"双非院校专业选择"决策模型

问题：现有分层决策模型主要覆盖985/211，对双非院校指导不足
解决：新增"双非院校专业选择模型"，强调行业匹配+地域就业
验证：经过3个真实案例测试，决策清晰度显著提升
```

### 3. 报告问题

**Issue模板**：

- [ ] 事实性错误（学校名称、政策信息等）
- [ ] 逻辑不一致
- [ ] 表述不清晰
- [ ] 信息过时
- [ ] 其他

#### 问题描述
[详细描述你发现的问题]

#### 所在位置
- 文件：`SKILL.md` / `README.md` / 其他
- 行号/章节：[具体位置]

#### 建议修改
[如果有，请提供你的修改建议]

#### 影响范围
- [ ] 轻微（措辞优化）
- [ ] 中等（影响理解）
- [ ] 严重（可能导致错误决策）

### 4. 分享对话示例

如果你使用这个Skill获得了有价值的对话，欢迎分享！

**步骤**：
1. 在 `examples/` 目录下创建新的Markdown文件
2. 文件命名格式：`XX-描述性名称.md`（例如：`06-art-student-strategy.md`）
3. 包含以下内容：
   - 对话背景（学生情况、问题场景）
   - 完整对话记录
   - 关键决策点分析
   - 对其他同学的参考价值

**示例文件结构**：
```markdown
# 艺术生跨专业选择案例

## 背景
- 学生：美术生，文化课500分
- 问题：不想继续学美术，能否选普通专业
- 省份：XX省，新高考3+1+2模式

## 对话记录

**学生**：我是美术生，但不想学美术专业了，能用普通分数填报吗？

**框框**：同学，这个问题要看你们省的政策...

[完整对话]

## 关键决策点
1. 艺术类vs普通类批次的关系
2. 文化课分数的双重价值
3. 提前批的风险规避

## 参考价值
适合所有艺术类考生面临跨专业选择的场景。
```

## 🔄 提交流程

### 第一步：Fork仓库

点击GitHub页面右上角的"Fork"按钮，将仓库复制到你的账号下。

### 第二步：克隆到本地

```bash
git clone https://github.com/你的用户名/kuangkuang-gaokao-perspective.git
cd kuangkuang-gaokao-perspective
```

### 第三步：创建分支

```bash
git checkout -b feature/你的功能描述
# 或
git checkout -b fix/你的修复描述
```

分支命名规范：
- `feature/` - 新功能或内容
- `fix/` - 修复问题
- `docs/` - 文档改进
- `refactor/` - 重构代码

### 第四步：进行修改

按照上述贡献方式进行修改。

### 第五步：提交更改

```bash
git add .
git commit -m "类型: 简短描述

详细说明（如需要）
"
```

提交信息类型：
- `feat`: 新功能
- `fix`: 修复问题
- `docs`: 文档修改
- `improve`: 改进现有内容
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具相关

### 第六步：推送到GitHub

```bash
git push origin feature/你的功能描述
```

### 第七步：创建Pull Request

1. 打开你Fork的仓库页面
2. 点击"Pull Request"按钮
3. 填写PR标题和描述：
   - 标题：简洁说明做了什么
   - 描述：详细说明为什么这么做、解决了什么问题
4. 提交PR等待审核

## 📋 开发规范

### Markdown格式规范

1. **标题层级**：
   - 一级标题（#）：文件标题
   - 二级标题（##）：主要章节
   - 三级标题（###）：子章节
   - 四级标题（####）：细节说明

2. **列表**：
   - 有序列表用于步骤说明
   - 无序列表用于并列项
   - 任务列表用于checklist

3. **代码块**：
   - 使用三个反引号包裹
   - 指定语言（bash、markdown等）

4. **强调**：
   - **粗体**：重点强调
   - *斜体*：次要强调
   - `代码`：专有名词、命令、文件名

5. **链接**：
   - 外部链接：`[显示文本](URL)`
   - 内部链接：`[显示文本](相对路径)`

### 内容编写规范

1. **语言风格**：
   - 简洁直接，避免冗长
   - 使用"你"而非"您"
   - 口语化但不失专业性

2. **示例提供**：
   - 重要概念必须有示例
   - 示例要具体、真实、有代表性

3. **边界说明**：
   - 明确适用范围
   - 诚实说明局限性
   - 不做过度承诺
